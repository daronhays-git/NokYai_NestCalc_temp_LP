# /handoff — Project Handoff Generator

## Purpose
Generate a structured handoff document for resuming work after a break or onboarding a collaborator. Pulls current state from live sources — git history, GitHub Issues, agent configs — not from other docs that may be stale. Saves the result to a dated file in `/docs/`.

---

## Files to Read Before Running
1. `CLAUDE.md` — dev conventions, commands, architecture summary
2. `project-brief.md` — what this product is, who it's for, monetization model, launch timeline
3. `module-map.md` — file structure, component inventory, backend functions, locked files
4. `docs/doc-inventory.md` — documentation coverage status and known gaps
5. `CHANGELOG.md` — most recent entry only (what shipped last)
6. `.claude/commands/` — list of active slash commands
7. `.claude/agents/` — list of active agent definitions
8. `.github/workflows/` — active CI workflows

---

## What Handoff Does

### Step 1 — Read Project Context
Read the four foundation files (`CLAUDE.md`, `project-brief.md`, `module-map.md`, `docs/doc-inventory.md`). From these, extract:
- One-paragraph product description (from `project-brief.md` Problem Statement + Core Value Proposition)
- Tech stack table (from `project-brief.md`)
- Active file structure patterns and key conventions (from `CLAUDE.md` and `module-map.md`)
- Known documentation gaps (from `docs/doc-inventory.md` §Coverage Gaps — high-priority items only)

### Step 2 — Read Current State from Git
Run the following commands:

```
git -C [PROJECT_PATH] log --oneline --no-merges -10
git -C [PROJECT_PATH] status --short
git -C [PROJECT_PATH] stash list
```

From these, determine:
- What shipped most recently (last 3–5 meaningful commits, excluding operational Beacon/Finance outputs)
- What is untracked or modified but not yet committed (in-progress work)
- Any stashed changes

Read the most recent entry in `CHANGELOG.md` for a categorized summary of recent work.

### Step 3 — Read Agent & Automation Status
List all files in `.claude/commands/`, `.claude/agents/`, and `.github/workflows/`. For each workflow file, read the `on:` trigger block to determine when it fires. Produce a brief inventory:

- **Slash commands:** one line per command — extract the purpose from the file's title line (`# /command-name — description`). Do not read the full file.
- **Agent definitions:** one line per agent in `.claude/agents/` — name and role. Note: most agents live in `.claude/commands/` as slash commands; `.claude/agents/` holds definitions for CI-invoked agents.
- **CI workflows:** use a table with columns: workflow filename, trigger (highlight `schedule:` cron triggers as **Scheduled**), and what it does.

Do not read any file beyond its title line (for commands) or `on:` block (for workflows).

### Step 4 — Check Known Issues & Priorities
Run:
```
gh issue list --state open --limit 20 --json number,title,labels,createdAt
```

If the command fails (no GitHub auth or no remote), note it and skip. Do not error out.

Check for a recent Beacon digest file matching `beacon-digest-*.md` or `beacon-daily.md`. If one exists and is less than 7 days old, read its summary section.

Check `CHANGELOG.md` for any `🐛 Fixes` in the most recent entry that suggest still-open problem areas.

### Step 5 — Resolve Output Path and Project Name
- **Default project name:** read from `project-brief.md`'s first heading
- **Argument override:** If the user passed an argument (e.g., `/handoff myproject`), use that as the project name
- **Output file:** `docs/handoff-[project-name]-[YYYY-MM-DD].md` using today's date

### Step 6 — Build and Write the Handoff Document
Assemble the document using the template below. Write it to the resolved path.

## How to Run

**Default — generate handoff for this project:**
```
/handoff
```

**With project name override:**
```
/handoff my-project-v2
/handoff client-abc
```

**Rules while running:**
1. Read live sources first — git log and git status are authoritative over any doc that describes "current state."
2. The handoff document is for someone who may not have run the project recently. Include exact commands, not paraphrases.
3. Mark anything that expires quickly with `(verify current status)` — this includes payment mode, env var values, and anything from a doc that hasn't been updated in the last 2 weeks.
4. If `gh issue list` fails, note "GitHub Issues: inaccessible — check manually at [repo URL if known]" rather than leaving the section blank.
5. Do not reproduce the full content of locked files or large config tables — summarize them.
6. When running git commands, always use `git -C [PROJECT_PATH]` instead of `cd <path> && git ...`.
7. If a `docs/handoff-[name]-[date].md` file already exists for today's date, overwrite it — reruns on the same day should refresh the document.
